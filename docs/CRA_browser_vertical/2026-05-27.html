A parting comment from Daniel Ehrenberg for us to consider, about the core challenge of this spec:

I might not have written it out very well before, but here’s how I see our core trilemma on how to write this specification: there is a tradeoff between three options for how to write requirements

  1. What the EC say they want: normative references. But web normative references are a vast recursive web and undated, contradicting their requirements. The first thing we would want to reference is the WHATWG HTML specification—is that permissible?
  2. Copying inline the contents of other specifications, with code-like assessment steps. HAS didn’t complain about this in the mature draft, but open consultation comments and other comments from Google, Mozilla and others have widely rejected this approach, and not just because the details were very inaccurate in the mature draft, but because even if updated, it freezes the web in its current state
  3. My proposed solution, documentation-based requirements which say that the implemented interface has to be documented and meet certain general requirements, and have tests for it. This contradicts the EC’s goal that things be executable, without referencing the web’s unversioned test suites.
