[15:57:22.0125] <rbuckton>
In November I asked that we postpone advancement until we had discussed cross-cutting concerns with pattern matching, but I don't believe that was ever addressed. In short, there is a fair amount of overlap between what's been proposed for Declarations in Conditionals and Pattern Matching's `is..let` syntax. If Decls in Conditionals advances to stage 2 or 2.7, it will likely force pattern matching into a direction that many of the champions have expressed opposition to. It may mean that we are forced to rely on existing binding/assignment pattern-like semantics such as I documented here: https://gist.github.com/rbuckton/df6ade207eecad4fc94cedc3aae79ceb. This could mean we would be back to needing something like `^` or `@` to designate whether something is a reference or a binding and `[]` syntax would not be exhaustive. Declarations in Conditionals also appears to be leaning towards only exposing the binding in the `if` branch, which is opposite of what we'd previously discussed for `let` patterns due to refactoring concerns.

This is the progression of the Declarations in Conditionals proposal, as I see it:

```
if (let x = a; x) ...; // separate declaration and condition have no conflict
if (let x = a) ...; // no condition (relies on truthiness) has no conflict, but begins the slippery slope
if (let { x, y } = a) ...; // very much crossing into pattern matching territory though not a concern yet.
if (let [x, y] = a) ...; // this would likely conflict with pattern matching which wants `[x, y]` to only match `[a, b]` and not `[a]` or `[a, b, c]`.
if (let Point(x, y) = a) ...; // already suggested by bakkot, in the issue linked above.

// as opposed to `is..let` patterns
if (a is let x) ...; // covers the first two cases
if (a is { x: let x, y: let y }) ...; // covers third case
if (a is { let x, let y }) ...; // shorthand form of previous line
if (a is [let x, let y]) ...; // covers fourth case, but semantics are stricter
if (a is Point(let x, let y)) ...; // covers fifth case
```

[16:10:33.0037] <rbuckton>
However, for Declarations in Conditionals to cover pattern matching, we'd likely need to implement things like `{| |}` for strict object matching, `[| |]` for strict array matching, no `let` patterns, and would need either `^` or `@` to distinguish between a reference and a binding, e.g.:

```
if (let { x: @Number, y: y@Number } = a) ...; // can't use `{ x: Number }` because that's just declaring a new binding named `Number`
if (let { x: @NaN, y: @Infinity } = a) ...; // how would we even do `-Infinity` unless its via `@Number.NEGATIVE_INFINITY`?
```

Destructuring syntax and semantics weren't designed for proper pattern matching, and shoehorning pattern matching in is going to be a huge wart on the language design, IMO.
