We really need to discuss the Declarations in Conditionals proposal, currently on next week's agenda for Stage 2 or 2.7, as I believe it could have ramifications for pattern matching, especially the `is` and `let`-patterns:

```js
// decls in conditionals
if (let p = obj.position; p instanceof Point) {
  ...
}

// pattern matching (mvp)
if (obj.position is Point and let p) {
  ...
}

// pattern matching (future potential)
if (obj is Point p) {
 ...
}
```

Even if it does not immediately impact pattern matching, I'm concerned it will push the committee towards shoehorning pattern matching into existing destructuring and binding patterns, which is an avenue we've already explored and rejected. For instance, if we support patternless `if (let x = y)`, it is a slippery slope towards `if (let [a, b] = ar)` or `if (let { x, y } = p)`, not unlike `if..let` in Rust. Especially if extractors were to advance independently of pattern matching. 

There's also https://github.com/tc39/proposal-Declarations-in-Conditionals/issues/3, which is suggesting binding semantics in opposition to the semantics for `let` patterns we've discussed previously. This could potentially force us into a specific path for that feature as well.

I think we should be considering the directions the Declarations in Conditionals proposal could take and form a position as to how pattern matching either aligns with or opposes this syntax prior to next week's plenary.
