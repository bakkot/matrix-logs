20:55<KiChjang>am i correct in thinking that WebIDL and ECMAScript are both different languages, and browsers just happen to use ECMAScript to implement WebIDL interfaces?
21:02<TabAtkins>KiChjang: Kinda. WebIDL was originally supposed to be an independent language for describing how interfaces should be implemented, so that you could do the actual implementation in multiple different target languages.
21:03<KiChjang>TabAtkins: so, would i be crazy if i think that wasm would allow other languages other than JS to implement WebIDL interfaces as well?
21:03<KiChjang>(once wasm is mature enough that is)
21:03<TabAtkins>But we abandoned that pretense a while back, because it turns out Java and JS have different execution semantics (who knew!) which changes how you want to shape the interfaces between the two languages. WebIDL is now just a more-succinct way of describing JS APIs that takes a lot of the busywork away.
21:04<TabAtkins>Other languages implementing APIs described by WebIDL will still run into the "other languages might not offer the same constructs in the same way" problem that Java/JS had.
21:17<KiChjang>hmm, so in other for other languages to substitute JS as the interface layer, they'll need to not only implement the APIs necessary, but also the execution model?
21:17KiChjangrecalls that there is a project to replace JS with ruby in the browser
21:39<annevk>KiChjang: wasm will impl IDL bindings
23:18<TabAtkins>KiChjang: At least a little bit. Like, you can't implement WebIDL's `iterable<Foo>` if your language doesn't have something closely equivalent to iterables.
