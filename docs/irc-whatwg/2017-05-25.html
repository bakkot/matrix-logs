08:40<hsivonen>(of course, I missing something *very* basic, but what?)
08:42<hsivonen>maybe what I want is learning async/await right away and skipping over how things were like before them
08:42<TabAtkins>hsivonen: If you just want the resulting promise to contain a value you obtain from doing stuff in the .then() callback, just return that value; it implicitly gets wrapped in a fresh Promise.
08:43<TabAtkins>If you the "stuff" you're doing ends up with you holding another Promise, and you want the result of the .then() to contain what's inside of your "other Promise", again just return it - it automatically chains the internal value.
08:44<hsivonen>TabAtkins: ooh. so magical.
08:45<TabAtkins>The only time you ever have to create a new promise manually is if you're not currently in a promise and want to start one. (Or you're using a legacy async api that doesn't return a promise, and want it to interoperate with a promise chain.)
08:45<TabAtkins>hsivonen: It's just a functor/monad. ^_^
08:45<hsivonen>TabAtkins: So I just "return;" from a .then() callback to return a promise than simply resolves?
08:45<TabAtkins>Yup.
08:45<hsivonen>TabAtkins: thanks
08:45<TabAtkins>If you just "return;" like that, then the resulting promise will resolve to undefined (your return value), but it will definitely resolve successfully.
08:46<TabAtkins>aka:
08:46<TabAtkins>let x = fetch(...).then(res=>doSomeWork()); => x is a promise holding the result of doSomeWork().
08:47<hsivonen>TabAtkins: thanks. that was what I was missing
08:47<TabAtkins>let x = fetch(...).then(res=>doSomeAsyncWork()); => x is a promise that will resolve when doSomeAsyncWork()'s promise resolves, to the same value.
08:48<TabAtkins>And if doSomeWork() throws, or the promise returned by doSomeAsyncWork() rejects, then x will reject as well.
