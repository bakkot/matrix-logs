21:06<TabAtkins>Basically browsers have a somewhat unintuitive performance profile that makes all the reverse-versions of the current 4 combinators pretty inefficient, even though they can be done fairly easily in selectors engines like Sizzle.
21:07<TabAtkins>It's because nearly all selectors matching in browsers happens on an incomplete DOM as a page is being loaded, while js-based selectors engines operate on complete doms.
21:09<TabAtkins>And the existing combinators happen to work great with that, as they only need information that is guaranteed to already be in the DOM.
21:09<TabAtkins>If you are trying to match div > span, when a span appears you are guaranteed that, if it has a div parent, you'll have already seen it and know about it because of the way HTML's structure works.
21:10<TabAtkins>While "span < div" doesn't offer the same promise - when you see a div, you then have to wait until all its children have been loaded before you can be sure if it has a span child.
