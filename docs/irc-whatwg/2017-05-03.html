12:06<KiChjang>what does "non-conforming" mean in the HTML spec?
12:10<MikeSmith>KiChjang: it generally means a document-conformance requirement
12:10<KiChjang>what does it mean for a document to conform?
12:12<MikeSmith>it means the document does not violate any requirements in the spec
12:13<MikeSmith>the document-conformance requirements are stated in order to prevent documents from using certain markup cases that have unintended bad effects
12:14<MikeSmith>see https://validator.w3.org/nu/about.html#why-validate for a short version
12:15<MikeSmith>see https://html.spec.whatwg.org/multipage/introduction.html#syntax-errors for a longer explanation
12:15<MikeSmith>and https://html.spec.whatwg.org/multipage/introduction.html#restrictions-on-content-models-and-on-attribute-values for yet more details
12:17<MikeSmith>but the shortest answer is:
12:17<MikeSmith>> There are some markup cases defined as errors because they are potential problems for accessibility, usability, interoperability, security, or maintainability—or because they can result in poor performance, or that might cause your scripts to fail in ways that are hard to troubleshoot.
12:18<MikeSmith>> Along with those, some markup cases are defined as errors because they can cause you to run into potential problems in HTML parsing and error-handling behavior—so that, say, you’d end up with some unintuitive, unexpected result in the DOM.
